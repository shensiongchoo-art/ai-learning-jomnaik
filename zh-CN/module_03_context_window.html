上下文窗口（context window）

为什么 LLM 有时会忘记你之前说过的话？来认识它的“记忆鱼缸”。

故事开始

第 1 幕 — 金鱼记忆问题
🐟
LLM 的记忆
你可能听过“金鱼只有三秒记忆”的说法。对真正的金鱼而言并不准确，但用来理解 LLM 却很形象。
LLM 的当前对话并没有无限记忆。每段对话都装在一个鱼缸里；鱼缸满了，旧消息就会被挤出去，LLM 也就看不见它们了。
核心概念：鱼缸 = 上下文窗口
第 2 幕 — 鱼缸里有什么？
💬
你的对话
对话中的每条消息都会占用鱼缸空间，包括你的消息和 LLM 的回答。就连一开始给出的指令也在其中。
你说（第 1 条消息）
“我叫 Sarah，是一名老师。”
LLM 回答
“你好，Sarah！很高兴认识一位老师。有什么可以帮你？”
两条消息都会占用鱼缸空间
第 3 幕 — 鱼缸满了
🌊
鱼缸溢出了！
经过许多轮对话，鱼缸装满了。最旧的消息，包括“我叫 Sarah”，被挤了出去。
你问（第 47 条消息）
“回答时可以称呼我的名字吗？”
LLM 困惑地回答
“我在目前看到的对话中找不到你的名字，可以再告诉我一次吗？”
它不是故意无礼，而是真的看不到那么久以前的内容！
第 4 幕 — 不同模型有不同大小的鱼缸
🪣
容量各不相同
不同 LLM 的上下文容量不同。有些只能容纳较短内容，有些则能处理整本书甚至更多资料。
容量以词元计量，还记得单元 2 的乐高积木吗？模型版本不断更新，因此使用前应查看该模型最新的官方限制。
鱼缸越大，能保留的对话内容越多

理解不同容量

上下文窗口容量的概念示意

产品限制变化很快；实际使用时请以对应模型的最新官方文档为准

容量层级适合处理需要注意
较小短对话、电子邮件、短文旧消息较快被挤出
中等报告、长对话、多篇文章仍需控制输入长度
较大书籍、大型代码库、资料包容量大不代表每个细节都同样受关注
超大多本书或大规模资料集合成本、延迟与检索策略依然重要

重点知识

关于上下文窗口要记住什么

  • 上下文窗口就是 LLM 的短期工作记忆，也就是它当前能够“看见”的全部内容
  • 你的消息和 LLM 的回答都会占用容量
  • 容量不足时，应用可能截断、摘要或舍弃较旧内容；具体行为取决于产品
  • 开始新对话通常意味着新的上下文，但某些产品会另行加入记忆功能
  • Claude Projects 等工具可以在对话之外保存说明或资料，但这和当前对话的上下文窗口并不是同一回事
